\begin{abstract}
We present BlackSwan, a Java bytecode decompiler implemented entirely in TypeScript that improves itself through a recursive feedback loop driven by an AI coding agent (Claude Code). The decompiler pipeline---bytecode parsing, intermediate representation, analysis, and source emission---was built by the AI agent itself across approximately two weeks, totaling ${\sim}32{,}000$ lines of TypeScript. We evaluate output quality using a composite metric drawn from nine similarity measures spanning textual, sequential, compression-based, and structural categories. In the recursive loop, the AI agent reads benchmark scores, reads the decompiler's own source code alongside competing decompilers' output, and proposes targeted patches. After iterative self-improvement, BlackSwan achieves a composite similarity score of 0.8693 on a 47-class benchmark, within 0.7\% of CFR (0.8760), a decompiler with over a decade of manual development. We discuss implications for automated deobfuscation, where the loop can be specialized against known obfuscation tools.
\end{abstract}

\begin{IEEEkeywords}
decompilation, reverse engineering, self-improving systems, large language models, Java bytecode, program analysis, AI-driven development
\end{IEEEkeywords}

\section{Introduction}
\label{sec:intro}

Java bytecode decompilation---the recovery of human-readable source code from compiled \texttt{.class} files---has been an active area of tooling for over two decades. Existing production decompilers such as CFR~\cite{cfr}, Procyon~\cite{procyon}, Vineflower~\cite{vineflower}, and FernFlower~\cite{fernflower} are predominantly maintained by small teams or individual developers. Each exhibits distinct failure modes, and progress is slow: a single obfuscation transform can be authored in a weekend, while adapting decompilers to handle it may take months.

We propose closing this gap by constructing a decompiler whose source code is iteratively modified by an AI coding agent in a closed feedback loop. The agent reads benchmark scores, reads its own source, reads competing decompilers' output on the same inputs, and proposes patches. The benchmark is re-run. The loop continues.

Critically, the AI agent also \emph{built} the decompiler. The entire pipeline---a port of OW2 ASM~\cite{asm} to TypeScript, an intermediate representation inspired by MapleIR~\cite{mapleir}, an analysis framework, and the decompiler itself---was authored by Claude Code (Opus 4.5) in approximately two weeks, comprising ${\sim}32{,}000$ lines of TypeScript. The language choice was architectural: the recursive loop requires the AI to read and edit the decompiler's source, and current-generation models are substantially more effective at modifying TypeScript than Java.

Our contributions are:
\begin{enumerate}
    \item A methodology for recursive, AI-driven self-improvement of a software tool via a multi-metric feedback loop with competitor-informed differential analysis.
    \item A composite evaluation framework of nine similarity measures designed to resist Goodhart's Law~\cite{goodhart} under optimization pressure.
    \item Empirical results showing an AI-built decompiler achieves 0.8693 composite similarity, within 0.7\% of CFR (0.8760) and 100\% file match rate, after iterative self-improvement.
    \item A discussion of specialization for automated deobfuscation via obfuscator-targeted loop configurations.
\end{enumerate}

\section{Related Work}
\label{sec:related}

\subsection{Java Decompilation}

Modern Java decompilers share a common architecture: bytecode parsing, IR construction, control flow analysis, type inference, and source emission. FernFlower~\cite{fernflower} uses an SSA-based IR with iterative dataflow analysis. CFR~\cite{cfr} employs aggressive pattern matching for control flow reconstruction. Procyon~\cite{procyon} introduced improved generic type recovery. Vineflower~\cite{vineflower}, a FernFlower fork, focuses on output readability. All are written in Java and have been developed over years to decades.

\subsection{AI-Assisted Decompilation}

Recent work uses LLMs directly as decompilers. LLM4Decompile~\cite{llm4decompile} fine-tunes models on paired assembly-source data. SLaDe~\cite{slade} uses sequence-to-sequence models for C decompilation. Nova~\cite{nova} applies LLMs to binary analysis broadly. These approaches perform inference-time translation---the LLM \emph{is} the decompiler.

BlackSwan is fundamentally different: the LLM modifies the decompiler's \emph{source code}, producing persistent improvements that apply to all future inputs without per-invocation inference cost.

\subsection{Self-Improving Systems}

Recursive self-improvement has been explored in program synthesis~\cite{levin2013, schmidhuber2003} and LLM-driven code generation~\cite{chen2024selfplay, shinn2023reflexion}. Reflexion~\cite{shinn2023reflexion} uses verbal feedback but operates at inference time. AlphaCode~\cite{alphacode} generates and filters solutions but does not modify its own implementation. BlackSwan targets the \emph{tool itself} for modification rather than individual outputs.

\label{sec:system}

BlackSwan is organized as a five-layer pipeline (Figure~\ref{fig:architecture}), with each layer implemented in TypeScript and editable by the AI agent.

\textbf{Layer 1: Bytecode Parsing.} A complete TypeScript port of OW2 ASM~\cite{asm} supporting all 200+ JVM opcodes, class file versions 45.3--65.0 (Java 1.1--21), and the full visitor-pattern API (\texttt{ClassReader}, \texttt{ClassWriter}, \texttt{MethodVisitor}, etc.). Round-trip correctness is verified: reading a \texttt{.class} file and writing it back produces byte-identical output. The port was completed in ${\sim}$4--5 hours of continuous Claude Code execution.

\textbf{Layer 2: Intermediate Representation.} A typed IR inspired by MapleIR~\cite{mapleir} that lifts JVM stack operations into expression trees within a control flow graph. A stack simulator walks bytecode instructions per basic block, popping operand expressions and constructing typed composite expressions (e.g., \texttt{ArithmeticExpr}, \texttt{VirtualInvocationExpr}). The IR comprises 19 expression node types and 13 statement types, with \texttt{PhiExpr} nodes at control flow merge points.

\textbf{Layer 3: Analysis \& Optimization.} Cross-class semantic analysis: class hierarchy graphs, virtual method table construction with diamond inheritance resolution, call graph building, and reference resolution. An extensible optimization pass framework supports fixed-point iteration; the initial pass is constant folding via worklist-based dataflow analysis.

\textbf{Layer 4: Source Code Emission.} Converts the analyzed IR to Java source. A structured control flow recovery engine attempts to reconstruct \texttt{while} loops, \texttt{if/else} chains, and \texttt{switch} statements from the CFG via pattern matching (natural loops, guard exits, join patterns, short-circuit boolean folding). When structured recovery fails, a fallback emitter produces a semantically-correct state-machine representation. Minimizing fallback frequency is a primary optimization target for the loop.

\textbf{Layer 5: AI Post-Processor.} An inference-time cleanup agent that renames auto-generated variables, fixes formatting, and removes decompiler artifacts. Architecturally distinct from the self-improvement loop: the loop modifies the decompiler's \emph{source code} (persistent), while the post-processor modifies \emph{output} (per-invocation).

\subsection{Design Decision: TypeScript}

The language choice was not a preference but an architectural constraint. The self-improvement loop requires the AI agent to read, reason about, and correctly edit the decompiler source. Current-generation models perform substantially better on TypeScript for this class of task. This required porting 12,000 lines of mature Java (OW2 ASM) to TypeScript---a significant investment that proved essential.

\section{Methodology}
\label{sec:methodology}

The core contribution is the methodology: a recursive feedback loop connecting an AI coding agent, a multi-metric evaluation harness, and an editable decompiler codebase. We describe each component.

\subsection{Recursive Self-Improvement Loop}

The loop (Figure~\ref{fig:loop}) operates in four phases per iteration:

\caption{Recursive self-improvement loop. Each iteration produces source code modifications that persist across all future runs.}
\label{fig:loop}
\end{figure}

\textbf{Phase 1 (Execute):} The benchmark harness runs BlackSwan and four established decompilers (CFR, Procyon, Vineflower, FernFlower) on an identical JAR, with 120-second timeouts per external process. All outputs are normalized through a Java formatter to eliminate superficial formatting differences.

\textbf{Phase 2 (Evaluate):} Each decompiled file is scored against its original source using nine similarity metrics (Section~\ref{sec:metrics}), producing per-file scores and aggregate statistics (mean, median, standard deviation, min, max). Results are emitted in JSON, Markdown, and tabular formats.

\textbf{Phase 3 (Analyze):} The AI agent (Claude Code, powered by Claude Opus) reads:
\begin{itemize}
    \item Per-file and aggregate benchmark scores
    \item The decompiler's own TypeScript source code
    \item Competitor decompilers' output on the same files
    \item Optionally, competitor decompilers' source code
\end{itemize}

The agent performs \emph{differential analysis}: identifying files where BlackSwan underperforms competitors, diffing outputs to understand the specific patterns handled differently, and tracing the cause to decompiler source code.

\textbf{Phase 4 (Patch):} The agent proposes and applies targeted modifications---new pattern recognizers, improved optimizations, bug fixes---then runs the test suite to verify correctness before committing.

\subsection{Competitor-Informed Differential Analysis}

A distinctive feature of the loop is that the AI agent has access to competing decompilers' output on the \emph{same inputs}. This enables a powerful feedback signal beyond aggregate scores: the agent can identify specific patterns where a competitor produces cleaner output, reason about the structural difference, and implement the corresponding heuristic in BlackSwan.

For example, in one iteration the agent observed that CFR collapsed switch statements where all cases end with \texttt{return} into a more readable form. It identified the relevant BlackSwan source file, understood the current fallback behavior, and added an equivalent pattern recognizer. This class of ``learning from the competition'' would be difficult to achieve with score-only feedback.

\subsection{AI Agent Configuration}

The agent has access to:
\begin{itemize}
    \item Full read/write access to all TypeScript source files
    \item Shell execution for building (\texttt{bun build}) and testing (\texttt{bun vitest})
    \item Benchmark execution and result inspection
    \item All competitor decompiler output files
\end{itemize}

No special prompting or fine-tuning is applied. The agent operates with its standard coding capabilities, using the benchmark scores and competitor output as natural-language-accessible feedback.

\section{Evaluation Framework}
\label{sec:metrics}

Designing the evaluation metric was critical: under optimization pressure from a capable AI agent, a single metric is vulnerable to Goodhart's Law~\cite{goodhart}---the agent may find degenerate solutions that maximize the score without improving actual quality. We address this with a composite of nine metrics spanning four mathematically distinct families.

\subsection{Metric Suite}

All metrics produce scores in $[0, 1]$ where 1 indicates identity with the original source. Table~\ref{tab:metrics} lists all metrics with their composite weights.

\begin{table}[t]
\centering
\caption{Similarity Metrics and Composite Weights}
\label{tab:metrics}
\footnotesize
\begin{tabular}{@{}llcl@{}}
\toprule
\textbf{Category} & \textbf{Metric} & \textbf{Weight} & \textbf{Foundation} \\
\midrule
\multirow{3}{*}{Text}
    & Jaccard Similarity & 0.08 & Set-theoretic \\
    & Cosine Similarity & 0.10 & Vector space \\
    & Dice Coefficient & 0.07 & Set-theoretic \\
\midrule
\multirow{3}{*}{Sequence}
    & Normalized Levenshtein & 0.12 & Edit distance \\
    & LCS Ratio & 0.12 & Subsequence \\
    & Line Diff Stats & 0.10 & Diff-based \\
\midrule
Compression
    & NCD & 0.11 & Information theory \\
\midrule
\multirow{2}{*}{Structural}
    & AST Similarity & \textbf{0.15} & Syntax tree \\
    & Control Flow Structure & 0.10 & CFG topology \\
\bottomrule
\end{tabular}
\end{table}

\textbf{Text metrics} measure token-level similarity. Jaccard computes $|A \cap B| / |A \cup B|$ over token sets. Cosine constructs term-frequency vectors: $\cos(\mathbf{a}, \mathbf{b}) = \mathbf{a} \cdot \mathbf{b} / (|\mathbf{a}| \cdot |\mathbf{b}|)$. Dice computes $2|A \cap B| / (|A| + |B|)$. These are order-insensitive, catching lexical similarity regardless of statement arrangement.

\textbf{Sequence metrics} measure ordering fidelity. Normalized Levenshtein computes $1 - d(a,b) / \max(|a|, |b|)$ via Wagner-Fischer DP. LCS Ratio normalizes the longest common subsequence length. Line Diff Stats computes the ratio of unchanged lines. These detect when tokens are present but misordered.

\textbf{Compression metric.} Normalized Compression Distance (NCD) approximates Kolmogorov complexity:
\[
\text{NCD}(x,y) = \frac{C(xy) - \min(C(x), C(y))}{\max(C(x), C(y))}
\]
using gzip at level 9. The similarity score is $1 - \text{NCD}$. This captures information-theoretic similarity independent of surface form.

\textbf{Structural metrics} measure code structure. AST Similarity uses tree-sitter~\cite{treesitter} to parse Java and combines node-type histogram cosine similarity (40\%), parent-child bigram Jaccard similarity (40\%), and depth profile Pearson correlation (20\%). Control Flow Structure combines control-flow keyword LCS ratio (40\%), nesting-depth Pearson correlation (30\%), and method-signature Jaccard similarity (30\%). These are the highest-weighted metrics (0.15 and 0.10 respectively), prioritizing structural correctness over surface-level similarity.

\subsection{Composite Scoring}

The composite score is:
\[
S_{\text{composite}} = \frac{\sum_{i} w_i \cdot s_i}{\sum_{i} w_i}
\]
where sums range over metrics that produced valid results. This graceful degradation handles parse failures (e.g., tree-sitter failing on malformed output).

\subsection{Resistance to Goodhart's Law}

The nine metrics span fundamentally different mathematical foundations:
\begin{itemize}
    \item Set-theoretic (Jaccard, Dice) vs.\ vector-space (Cosine)
    \item Sequential (Levenshtein, LCS) vs.\ unordered (text metrics)
    \item Information-theoretic (NCD) vs.\ syntactic (AST, Control Flow)
\end{itemize}

A degenerate solution that maximizes one family (e.g., emitting all reference tokens to spike Jaccard) would produce detectably low scores in others (e.g., collapsed Levenshtein from wrong ordering, low AST similarity from broken structure). The composite score requires concurrent improvement across all families---a much harder optimization target to game.

\subsection{Statistical Reporting}

For each decompiler, the harness reports per-file breakdowns and aggregate statistics: mean, median, standard deviation, min, max. This enables the AI agent to identify \emph{specific} weak files and \emph{specific} lagging metrics, directing its analysis effort rather than operating from aggregate feedback alone.

\section{AI Post-Processor}
\label{sec:postprocessor}

Independently of the self-improvement loop, BlackSwan includes an inference-time cleanup agent. After decompilation, a second AI agent processes each output file with a tightly constrained prompt:

\begin{quote}
\emph{``Preserve logical behavior. Fix formatting. Rename auto-generated locals. Fix obvious decompiler artifacts. Do NOT add comments, documentation, annotations, methods, fields, or classes.''}
\end{quote}

The agent processes files sequentially, signaling completion via an MCP~(Model Context Protocol) \texttt{file\_done} tool. This provides two independent improvement axes: the decompiler gets structurally better through the loop (persistent), while the cleanup agent catches cosmetic artifacts at runtime (per-invocation). The benchmark evaluates both configurations, measuring the marginal contribution of each.

\section{Applications to Automated Deobfuscation}
\label{sec:deobfuscation}

The recursive loop has direct implications for reversing obfuscated software.

\subsection{Obfuscator-Specialized Variants}

Given paired samples (original source $\leftrightarrow$ obfuscated output) from a known obfuscator, and optionally the obfuscator's source code, the loop can be configured with the obfuscated-vs-original pairs as the benchmark. The AI agent writes targeted deobfuscation passes that reverse specific transforms---string decryption, control flow de-flattening, reflection proxy removal---validated against the paired ground truth.

The result is a decompiler \emph{specialized} for that obfuscator: not a general-purpose tool handling obfuscation heuristically, but a targeted pipeline with transform-specific inverse operations.

\subsection{Malware Family Analysis}

When malware samples share an obfuscation toolchain, the loop specializes against the common obfuscator. New samples from the same family are decompiled by the specialized variant, producing substantially cleaner output than general-purpose decompilers.

\subsection{Obfuscator Fingerprinting}

Composite scores from obfuscator-specialized variants serve as fingerprints. A sample's relative score across variants indicates which obfuscation tool was applied, enabling automated attribution: ``This sample scores 0.3 against our Allatori-tuned variant but 0.7 against ZKM-tuned''---identifying the obfuscator without manual reverse engineering.

\section{Discussion}
\label{sec:discussion}

\subsection{Persistent vs.\ Inference-Time Improvement}

A key distinction is that loop improvements are \emph{persistent}---source code modifications committed to the codebase, applied to all future inputs without additional inference cost. This contrasts with inference-time approaches (including our own AI post-processor and prior work such as LLM4Decompile~\cite{llm4decompile}) where each invocation incurs compute cost and may produce inconsistent results. The loop amortizes AI compute over all future decompilation runs.

\subsection{The Role of Competitor Access}

Providing the agent with competitor output created a uniquely powerful feedback signal. Beyond knowing \emph{that} it underperformed, the agent could see \emph{how} the competitor handled the same input, reason about the structural difference, and implement the corresponding strategy. This differential analysis is more information-rich than score-only feedback and could be applied to other domains where reference implementations exist.

\subsection{Limitations}

\textbf{Exception handling}: Structured recovery currently rejects methods with exception handlers, falling back to state machines. This is the largest single contributor to the quality gap with Vineflower.

\textbf{Benchmark scale}: Our 47-file benchmark spans diverse features but is limited in size. Evaluation on larger, real-world codebases (e.g., Apache Commons, Guava) is needed.

\textbf{Convergence}: Improvements diminish after ${\sim}$5 iterations. Whether this reflects architectural limits or the agent's reasoning capacity is an open question.

\textbf{Regression risk}: Patches occasionally improve some files while degrading others. Human oversight is required for accept/reject decisions. Automated regression detection via per-file score tracking partially mitigates this.

\textbf{Reproducibility}: AI agent behavior is non-deterministic. Different runs of the loop may produce different modifications, though we observe convergence to similar score ranges.

\section{Conclusion}
\label{sec:conclusion}

We have presented BlackSwan, a Java decompiler built and iteratively improved by an AI coding agent. The methodology---a recursive loop connecting an AI agent, a multi-metric benchmark, and an editable codebase---produces measurable, persistent improvements in decompiler quality, achieving competitive scores against tools with decades of manual development.

The key enablers are: (1) implementing the pipeline in a language the AI edits effectively, (2) a composite metric spanning nine mathematically distinct similarity measures to resist gaming, and (3) competitor-informed differential analysis providing richer feedback than scores alone.

The loop architecture generalizes to obfuscator-specialized decompilation, where paired samples and obfuscator source code enable targeted deobfuscation pass generation. This direction has significant implications for malware analysis and software security: the economic asymmetry between obfuscation (cheap, mechanical) and decompilation (expensive, analytical) may be fundamentally altered when the decompiler can recursively adapt.

All source code is available at \url{https://www.blackswan.sh}.
